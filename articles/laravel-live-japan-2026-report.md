## はじめに

![立川ステージガーデン前のLaravel Live Japan看板](/images/laravel-live-japan-2026-report/venue.jpg =400x)

2026年5月26日・27日に開催された「Laravel Live Japan 2026」に、現地参加してきました。本記事は、セッションの網羅的なまとめではなく、各発表を聴きながら自分が何を考え、どこに悩み、どこに違和感を抱いたかを正直に書き残した、参加者としてのレポートです。

セッションの正確な内容はアーカイブで確認していただけます。本記事では、自分の頭の中で起きていたことの方を、できるだけそのまま記録に残しておきたいと思います。

- 公式サイト: [Laravel Live Japan](https://laravellive.jp/ja)
- 日時: 2026年5月26日(火)・27日(水)
- 場所: 立川ステージガーデン
- 参加人数: 500名前後（うち40%が海外からの参加者）
- アーカイブ: [Day1](https://www.youtube.com/watch?v=TR25AkhjiRc) / [Day2](https://www.youtube.com/watch?v=VbM_o0adyoc)

参加人数まわりは主催の濱崎さん[@avosalmon](https://x.com/avosalmon)のポストが詳しいです。最終的には当日600名近くまで増えたみたいですが、目標だった500人を達成し、そのうち約4割が海外、40カ国以上から集まったとのことでした。

https://x.com/avosalmon/status/2053797966230831604

登壇者・スタッフ・スポンサー、そして会場で話してくださったすべてのエンジニアのみなさんに、まず感謝を伝えさせてください。本当に良いイベントでした。

## 2日間を通して受け取ったもの

セッション個別の話に入る前に、頭の中に残った「全体としての感触」を残しておきます。

ひとつは、**Laravel というフレームワークが「これから自分たちはこうありたい」と、はっきり方向を示してきている**ということ。AI SDK、Laravel Boost、Laravel Cloud、Inertia.js、Strict系のデフォルト ── 個々のトピックは別物に見えても、根っこに「AI時代に開発者が安心して書けるフレームワークでありたい」という意思を感じました。

もうひとつは、**自分は思っているよりもまだ Laravel のことを知らない**、ということ。後述する武田さんの発表で「使ったことのないコマンドは一度叩いてみてほしい」という話があったのですが、これは自分にとってかなり耳が痛い言葉でした。デフォルトで用意されているものを知らずに、似たような自前実装を抱えている部分が、自社のプロダクトには確実にあります。

そして、もうひとつ大切に持ち帰ったのが、**「やっぱり現地に出る価値はある」という体感そのもの**でした。これは内容よりも、体験として強く残っています。詳しくは後半に書きます。

## 印象に残ったセッション

### Taylor Otwell — Keynote

![Taylor Otwellのスピーカー紹介スライドと、演台に立つTaylor](/images/laravel-live-japan-2026-report/taylor-session.png =500x)

Taylor のキーノートは、正直に言うと「ここで体が前のめりになった」みたいな特定の瞬間はありませんでした。それよりもむしろ、**「Laravel はこういう方向に進化していきたいんだ」というメッセージ全体を、フラットに浴び続けた1時間**、という感覚に近いです。

Laravel Boost が `agents.md` を吐き出して `/upgrade` でバージョンアップを走らせる流れ、Laravel AI SDK のサブエージェント。どれも単体で見れば派手な機能なのですが、自分にとっては「個別の機能を覚える」よりも、**この方向性を生んでいる人たちが本気でこれを推している、その温度感を体感する**時間だったように思います。

Laravel Cloud の東京リージョン正式ローンチは、業務寄りの観点では一番現実味のあった発表でした。「個人的に何か作ろうと思ったとき、これで全部済むのか」と素直に思いました。

15年続いているフレームワークの創設者が、今も自分の手でデモを打ち、自分の言葉でプロダクトを語る。Taylor のその姿勢を生で見られたこと自体が、何より大きな収穫だった気がします。

### Nuno Maduro — "Strict AI Engineering"

Nuno のセッションは、本イベントの中で最も**「ああ、自分の方向性は間違っていなかった」と確認できた発表**でした。

![Nuno Maduroの登壇シーン。スクリーンに型付きのコードが映る](/images/laravel-live-japan-2026-report/nuno-session.jpg =400x)

> **"Agent coding is not vibe coding."** — Nuno Maduro

このフレーズは、以前から自分も意識していたものでした。AIに任せて気持ちよく書けば書くほど、明確な制約を入れておかないとプロダクトはすぐに崩れる ── そういう感覚は、自分の中にすでにあった。だからこのセッションは、新しい気づきを得る場というより、**自分の中の仮説に、Nuno という強い裏付けが乗った**ような体験でした。

そのうえで、**一番驚いたのは Strict 系設定の手軽さ**でした。`Model::shouldBeStrict()` や `prohibitDestructiveCommands()` のように、Laravel 側がすでに「ここをガードしておきたいよね」というレールを用意してくれている、というのは恥ずかしながら知らなかった。1行で N+1 を防げて、1行で本番での破壊的コマンドを止められる。**「フレームワークがここまでお膳立てしてくれているなら、これは使うしかない」**と素直に思いました。

一方で、**全面的に賛同しきれなかった部分もあります**。それは「コードカバレッジ100%」を強く推す姿勢です。AIで生成されるコードが増える今、テストを徹底するという発想自体は理解できます。ただ、自分の実感としては、100%はやはり費用対効果が見合わないケースが多い。書くこと自体に時間がかかるうえに、制約がかかりすぎて、変更スピードが落ちる場面も出てくるはずです。

これは**新規プロダクトには取り入れたいが、既存プロダクトに丸ごと持ち込むのは現実的でない**、というのが自分の今の結論です。新しいプロダクトを始めるときの「最初から効かせるルール」として一番効くものだと思います。

### 武田憲太郎さん — "Why Laravel Apps Break"

本イベントで一番、自分のプロダクトと正面から向き合わされたのが、武田さんの発表でした。

**「AIに選択肢を与えるほど予測困難になり、生成されるコードの品質が下がる」** ── これは、自分の中の薄々の違和感を、見事に言語化してくれていました。AIに任せる前提でアーキテクチャを考えるなら、選択肢は最初から少ないほうがいい。Laravel Way に従うとは、そういうことだったのか、と腑に落ちる感覚がありました。

そして、ここからが正直なところなのですが ── **このセッションで、自分は古傷を抉られた気がしました**。

自社プロダクトには、DDD の思想を一部取り入れた Repository が存在しています。実は、ここ最近**「Repository、もう要らないんじゃないか」**という議論を社内でしていて、ちょうど答えを出そうとしていたタイミングでした。

成熟したプロダクトのリニューアルを1年ほど進めていて、残りも半分を切ったところ。今からやめるのも怖くて、最終的には「渋々残す」という方向で着地させたばかりだったんです。

武田さんが名指ししていたのは Repository というより、「DDD やクリーンアーキテクチャ、独自レイヤーのような選択肢を増やすほど Laravel Way から外れ、AI にとっても扱いづらくなる」という、もっと広い話でした。でも、それを自分のプロダクトの Repository に引きつけて聞いているうちに、**「やっぱりやめたほうがいいんじゃないか」という思いが、再燃してしまいました**。

正直、いまもう一度同じ議論を社内でできるかというと、難しい部分はあります。それでも、心の中ではずっと「やめたい」と思い続けることになりそうです。今回の発表で簡単に判断が変わるわけではないけれど、判断の重さは確実に増しました。

とはいえ、冷静になると話はそう単純でもなくて。自社プロダクトはそれなりに長く動いてきて、設計もある程度枯れています。なので「AI に無限の選択肢を渡してカオスになる」みたいな状況とは、実はちょっと違うんじゃないかとも思っているんです。毎回ゼロから書き方を迷うようなフェーズは、とっくに過ぎている。

それでも Laravel Way に寄せたいと思うのは、別の理由からです。いまちょうど進めているリニューアルは、Laravel だけの構成から、バックを Laravel API、フロントを Next.js に載せ替えるという内容なんですが、こういう大きな作り直しのタイミングでこそ、プロダクト独自の書き方を引きずるより、素直に Laravel Way に則った形へリファクタリングしておきたい。そのほうが、あとから新しいエンジニアが入ってきたときにも、機能を足していく拡張性の面でも、後々ラクになるはずです。「AI のために」というより、「未来の自分たちと、これから入ってくる仲間のために」寄せておきたい、という感覚に近いかもしれません。

もうひとつドキッとしたのが、自社プロダクトでは Event や Listener といった Laravel デフォルトのクラスをほとんど使えていない、という事実です。「副作用は Event & Listener に分離する」というのは、聞けば当たり前なのに、自分はそもそも選択肢として持っていなかった。**いきなり全面導入しなくていいので、まずは「使えるところから探りたい」**と思っています。

ちなみに、武田さんの「最後の防波堤は人間」というメッセージは、印象的なフレーズではあるのですが、自分にはそこまで強く刺さりませんでした。たぶん、AI への向き合い方として「最後に人間が責任を取る」というのは、自分の中ではある程度すでに当たり前の感覚になっていたからだと思います。

刺さらなかった部分も含めて、自分の今の現在地が整理できた、価値の濃いセッションでした。

スライド: [Why Laravel Apps Break - Mastering the Fundamentals to Keep Them Maintainable](https://speakerdeck.com/kentaroutakeda/why-laravel-apps-break-mastering-the-fundamentals-to-keep-them-maintainable)

### Colin DeCarlo — "The New Bottleneck in Software"

Lightning Talk の枠で語られた内容ながら、自分の中で一番「ぐうの音も出ない」と思った発表でした。

> AI 後のボトルネックは、コーディングではなく「仕様定義」と「コードレビュー」になる。

Colin のこの主張に対して、自分は素直に「両方とも、自分の最近の仕事のボトルネックそのものです」と認めるしかありませんでした。**仕様の解像度が甘いまま AI に投げて、出てきたコードのレビューに時間が溶ける**。書く時間より、設計と判断の時間の方が確実に重くなっている。

短いトークだったぶん、メッセージの密度が高く、しばらく自問し続けることになりそうです。「AI で速くなった」と言っている裏側で、本当に効いているところに自分は時間を使えているのか。

### Joe Tannenbaum — "The Modern Monolith, Revisited"

Joe のセッションは、東京駅の列車時刻表アプリをライブで作っていくスタイルで、見ていてただ単純に楽しかったです。Optimistic Updates、Deferred Props、Polling、`<Form>` コンポーネント。**Inertia.js v3 の体験そのものが、現場の不満をピンポイントで解いている**のがよくわかりました。

![Joe Tannenbaumのスライド「PAIN POINTS／問題点」。CORS、Two Codebases、Auth Complexity、API Drift の4つが並ぶ](/images/laravel-live-japan-2026-report/joe-pain-points.jpg =400x)

スライドに並んでいた「CORS」「Two Codebases」「Auth Complexity」「API Drift」。これ全部、Next.js + Laravel で日々向き合っている課題そのものでした。

そして、自分が一番欲しいと思ったのは Laravel Wayfinder。PHP のコードから TypeScript の型定義を自動生成してくれるツールです。

> サーバーとクライアントの不整合が起きたとき、赤く光ってほしい。

自社プロダクトは Laravel + Next.js 構成で、フロントとバックの間で型を共有する大変さを、自分は身をもって毎日感じています。今は API のレスポンスを見ながら、TypeScript の型を手で書いている。**「これ、もっと機械にやらせる方法あるよね？」という素朴な不満に、Wayfinder が真っ直ぐ答えてくれていた**感覚でした。

React ベースの Inertia.js は Next.js 構成にそのまま持ち込めないため、今すぐ自社で導入するのは現実的ではありません。それでも、**次に何か作るときの技術選定の候補として、確実に上位に入った**と思います。「モダンモノリス」という言葉も、改めて好印象でした。シンプルさを選び直すこと自体に、勇気がいる時代だなと感じます。

## 会場で起きたこと

ここまでセッションの話を書いてきましたが、自分が今回のイベントで一番強く持ち帰ったのは、実はセッションの中身ではありませんでした。

それは、**懇親会とランチマッチングで話した参加者との会話**です。

海外の方が本当に多くて、英語は全くと言っていいほど話せない自分にとっては、最初はかなり緊張する時間でした。それでも、ボディランゲージとパッションを総動員して、なんとか会話を続けようとする。会話できたのか、できなかったのかは、正直よくわかりません。それでもその時間が、**「もっと英語を勉強したい」と本気で思える刺激**になりました。

ここで自分が思った「英語を勉強したい」は、ふんわりした目標ではなく、わりと具体的なものです。

- **同時通訳なしでセッションを聴きたい**。今回の英語セッションはイヤホンを通して聴いていましたが、生の言葉と訳の間にはどうしても距離がある。スピーカーの選んだ単語のニュアンスを、直接受け取りたい。
- **仕事やキャリアのツールとして英語が欲しい**。海外コミュニティへの参加や、もしかしたら将来の選択肢を広げるためにも、ここで身につけておきたい。

それから、**いろんな立場・国籍・背景の人と話せたことそのもの**が、純粋に楽しかったです。社内だけで仕事をしていると、知識も価値観もどうしても閉じていきます。「えっ、それうちはまだやれてないな」「逆にうちはここを大事にしてる」と、自分の現在地が立体的に見える時間でした。

人見知りな自覚はあるのですが、最初の一杯で乾杯した相手と、気づけば1時間話している、ということが今回も起きました。最初に話すネタを1つだけ持っておくと、案外なんとかなります。私のおすすめは「今日のカンファレンスで面白かったセッションは？」です。これは万能です。

## これからのこと

イベントの余韻が消えないうちに、自分が「こうしていきたい」と思っていることを残しておきます。

技術寄りで取り組みたいこと：

- 自分のプロダクトに **Laravel Boost** を入れて、AI 向けの規約を整える
- `Model::shouldBeStrict()` を有効化し、**PHPStan / Larastan** を CI に乗せる
- **Wayfinder の思想**を参考に、Next.js + Laravel 構成での型同期手法を比較検証する
- Laravel Boost に含まれる設計ガイドを読み、**Laravel Way を学び直す**
- Event/Listener を、まずは「使えるところから探る」スタンスで導入していく
- Repository の去就は、もう一度自分の中で考え直す（たぶん、また悩む）

姿勢の話としてやっていきたいこと：

- **社外のエンジニアとの接点を、今より増やす**
- Laravel コミュニティに限らず、**他のコミュニティにも足を伸ばす**
- 次回 Laravel Live Japan が開催されるかは未定とのことですが、**開催されたら絶対に行く**
- **英語の勉強**（同時通訳なしでセッションを聴ける自分になる）

## おわりに

Laravel Live Japan 2026 は、自分にとって**「答えをもらう」イベントではなく、「悩み直す」イベント**でした。

Repository の選択を再考したくなり、AI とのつきあい方を確かめ直し、英語が話せない自分を突きつけられ、そして「やっぱり現地に出てよかった」と思える時間をもらいました。

セッションを作ってくれた登壇者のみなさん、運営してくださったみなさん、会場で話してくださったみなさんに、改めて心からの感謝を。

来年も、現地で会えるのを楽しみにしています。
